# Concept Emergence: Evolving Connections into a Knowledge Topology

Status: Draft v2 (revised — concepts merged into tags)
Date: 2026-05-13
Scope: Evolve the static article/highlight similarity feature into a semantic-tag-first knowledge discovery system. No new concept entity — extend the existing tag system to carry semantic intelligence.

---

## 1) Why this document exists

The current "connections" feature exposes cosine similarity scores between articles and highlights. It works technically but fails as a product: users see "73% similar" without understanding why, can't act on the information, and don't know what to do next. This plan defines a north-star vision, a phased roadmap to get there, TDD methodology for building it, and the subagent orchestration pattern to use during the building session.

---

## 2) Current state — what connections actually does

Three separate surfaces exist, all powered by pgvector cosine similarity:

**Article-level similarity (`/search/{item_id}/similar`)**
- Raw cosine similarity on item-level embeddings (1536-dim, text-embedding-3-small)
- Exposed via "Find Related" toggle at the bottom of each article in the reader
- Renders as a grid of articles with thumbnail + "X% similar" label
- Fixed threshold: 0.5. Uses only item-level embedding — chunk embeddings are NOT used here. No ranking beyond score.

**Highlight-to-highlight connections (`ConnectionsPanel`)**
- Finds highlights from other articles whose embeddings are similar
- Desktop-only (≥1280px), keyboard shortcut `c`
- Grouped by connected article with "Your: / Their:" label format
- Threshold: 0.5, limited arbitrarily to 5 connections per highlight

**Hybrid search (`/search/semantic`)**
- Routes queries through keyword/semantic/RRF fusion
- Chunk-level embeddings ARE used here — but not in the two surfaces above
- Returns articles + highlights as separate result sets

**Existing tag system:**
- `tags` = user-confirmed tag strings (ARRAY(String(100)))
- `auto_tags` = AI-suggested tags pending user accept/dismiss (ARRAY(String(100)))
- Tags generated by: embedding similarity to other tagged articles (free pass), then gpt-4o-mini fallback
- Tags are currently coarse categorical labels — "ai", "product", "startups"
- Tags have no individual embeddings, no salience scores
- Tag extraction does not read article content directly — it matches against the article's embedding similarity to other articles the user has already tagged

**Infrastructure that exists but is underutilized:**
- `content_chunks` table: 350-token passages each with a separate embedding
- Celery pipeline: embeddings generated for all articles, highlights, and chunks
- Redis: query embedding cache with 1hr TTL

## 3) Problem framing — why connections feel gimmicky

### The "so what?" gap

The root problem is that connections are ARTICLE-FIRST and SCORE-ONLY. The feature tells users "these two articles are 73% similar" without answering:
- **What idea** connects them?
- **What do I do** with this information?
- **What am I missing** that I haven't noticed?

### Why tags haven't fixed this yet

Tags were designed to be the semantic unit — the idea layer above raw similarity. But the current `generate_tags` pipeline drifts toward coarse categorization because it learns by similarity to existing tags the user has already confirmed. If a user's library is mostly tagged "ai" and "product", new articles get those same tags. It mirrors the user's current vocabulary rather than extracting what the article actually discusses.

Result: two articles both tagged "ai" can be about completely unrelated ideas. Tags exist but don't carry the semantic specificity needed for connection explanation.

### Real user failure modes today

| Scenario | What happens today | What should happen |
|----------|-------------------|--------------------|
| User reads about AI alignment, has 30 related saves | Click "Find Related", see a list, feel vaguely satisfied | "You have 8 saves on AI alignment. Key tension: deceptive alignment vs. corrigibility. You haven't read anything on [gap]." |
| User highlights a claim about LTV payback | ConnectionsPanel shows similar highlights | "Your 3 highlights on LTV relate to 2 themes: early payback models and PLG." |
| User starts a draft about distributed systems | No integration exists | Draft workspace surfaces: "You have 12 reads relevant to this topic. 5 unread." |
| User wants to understand their reading shape | No overview exists | A concept map shows reading clusters, bridges, and gaps |

### Why the gimmick feeling persists

1. **No semantic explanation** — "73% similar" has no meaning for users
2. **Tags too coarse** — "ai" doesn't explain that both articles discuss "mesa-optimization"
3. **No synthesis** — seeing a connection produces nothing
4. **Passive discovery only** — you must navigate to an article and click
5. **No temporal signal** — an article from two years ago is weighted identically to yesterday's
6. **Chunk embeddings unused in similarity** — fine-grained matches hidden behind item-level average
7. **Highlight cap is arbitrary** — 5 per highlight is meaningless

## 4) External patterns and lessons

### Readwise Reader
- **Pattern**: "Daily Review" resurfaces highlights using a spaced repetition algorithm
- **Why it works**: Converts static highlights into an active memory loop
- **sed.i implication**: A resurface mechanic for highlights (daily prompt, contextual cards) converts passive saves into active memory
- **What not to copy**: Full SRS scheduling is high-maintenance. Start with simpler "daily highlight" surface.

### Obsidian Graph View
- **Pattern**: Visual node graph where clusters emerge from links
- **Why it works**: Topology reveals reading shape — which topics are dense, which are isolated
- **sed.i implication**: A tag-cluster view showing reading themes answers "what am I actually reading about?"
- **What not to copy**: Full graph UX is complex. Start with a card grid of themes.

### Notion AI / Perplexity "related content"
- **Pattern**: Sidebar showing related documents at the point of use, explained in plain language
- **Why it works**: Connection appears AT THE POINT OF NEED, not as a separate feature
- **sed.i implication**: Draft workspace should surface relevant reading inline as you write

### Roam Research bidirectional links
- **Pattern**: Every concept reference creates a backlink in both directions
- **Why it works**: Concepts become first-class objects, not just search keywords
- **sed.i implication**: Semantic tags (fine-grained, content-extracted) stored with embeddings create the bidirectional linking behavior without manual effort

## 5) The design decision: extend tags, don't add concepts

Tags were always intended to be the semantic unit. The plan does not introduce a `content_concepts` table or a separate concept entity. Instead:

**`auto_tags` becomes the semantic tag store.** The `generate_tags` task is extended to extract fine-grained semantic labels directly from article content — not just by similarity to existing tags. Labels like "deceptive alignment", "mesa-optimization", "LTV payback period" live in `auto_tags` alongside or replacing coarse labels.

**A `tag_embeddings` lookup table is added.** This is infrastructure, not a new user-facing entity. It's a cache: `{label → Vector(1536)}`. Unique tag labels get embedded once and reused. This enables tag-level semantic similarity (finding articles that discuss similar ideas even without exact tag matches) and powers clustering.

**`tags` stays as user-confirmed labels.** Users continue to accept/dismiss from `auto_tags`. The UX is unchanged. The intelligence is in what `auto_tags` produces.

This approach:
- Reuses existing pipeline, schema, and UI without a new entity type
- Extends the tag system to do what it was designed to do
- Keeps one mental model: tags are the semantic unit, period

## 6) The north star vision

**One sentence:** sed.i should tell you not just which articles are similar, but which specific ideas connect them, how those ideas cluster across your reading history, and what you're missing.

**End-to-end user flow (north star):**

1. User saves 50 articles over a month across AI, product strategy, and distributed systems
2. Enhanced `generate_tags` extracts fine-grained labels: "deceptive alignment", "mesa-optimization", "LTV payback", "Paxos", "eventual consistency"
3. A weekly job clusters those tag embeddings into reading themes per user
4. User opens "Reading Themes" — sees 4 clusters: [AI Safety] [PLG] [Consistency Models] [Bridge: AI + infrastructure]
5. User clicks "AI Safety" — sees articles, their key highlights, the specific tags that link them
6. User starts a draft about AI alignment → draft workspace surfaces "12 reads on this theme, 5 unread"
7. User finishes draft → can cite reads with one click

**What this feature actually solves:**
- **The knowledge inventory problem**: "What do I actually know about topic X from my reading?"
- **The synthesis gap**: "I've read a lot, but I haven't made sense of it yet."
- **The writing friction problem**: "I know I saved something relevant but can't find it while I'm writing."

## 7) Gap model

### Capability matrix

| Capability | Today | Target | Confidence |
|-----------|-------|--------|------------|
| Article-level similarity | ✅ Works, score-only | Tag-explained similarity | High — just needs richer tags |
| Highlight connections | ✅ Works, arbitrary cap | Tag-grouped connections | High — needs tag layer |
| Semantic tag extraction | ❌ Coarse only | Fine-grained tags from article content | High — straightforward LLM task change |
| Tag embeddings | ❌ Missing | `tag_embeddings` lookup table | High — simple extension |
| Reading themes | ❌ Missing | Tag cluster UI page | Medium — cluster quality TBD |
| Draft ↔ reading integration | ❌ Missing | Contextual reading surface in draft | Low — complex UX |
| Connection feedback loop | ❌ Missing | "Was this useful?" | Low priority |
| Temporal ranking | ❌ Missing | Recency bias in similarity results | Medium |

### Where sed.i is already stronger
- Production-grade Celery pipeline (retries, dead-letter handling, status tracking)
- Multi-vector chunk embeddings already generated for every article
- MCP server: tag data immediately accessible to Claude sessions
- Auth + user scoping: all data user-scoped, no cross-user leakage risk

### Where sed.i is behind
- Tag extraction is coarse (learns from existing tags, not from article content)
- No tag-level embeddings (tags are just strings, no semantic indexing)
- No proactive surfacing (everything is user-triggered)
- Draft integration entirely missing

### Missing instrumentation
We cannot measure today which connections users click or whether connections change reading behavior. Phase 1 must include basic click telemetry before making data-driven threshold decisions.

## 8) Phased roadmap

### Phase 1 (Weeks 1–2): Semantic tag extraction

**Goal:** `auto_tags` contains fine-grained semantic labels extracted from article content. Connections show shared tags instead of raw scores. Tags have embeddings via a new lookup table.

**Build:**
- `tag_embeddings` table: `{id UUID PK, label TEXT UNIQUE, embedding Vector(1536), created_at}`
- Enhance `generate_tags` task: add a content-extraction pass that reads `title + description + summary` and calls gpt-4o-mini to extract 3-5 fine-grained semantic labels (e.g. "deceptive alignment", not "ai"). These go into `auto_tags` alongside or replacing coarse labels.
- After writing new tags to `auto_tags`, upsert each label into `tag_embeddings` (embed if not cached; reuse if already present)
- Backfill task `backfill_semantic_tags` for existing articles (rate-limited 50/min)
- Update `/search/{item_id}/similar` response: include `shared_tags: list[str]` — tags present on both source and result article
- Update "Find Related" UI cards: show tag pills below each result ("Shared ideas: AI alignment · mesa-optimization")
- Basic telemetry: `POST /search/telemetry` — `{surface, item_id, shared_tag, action: "click" | "dismiss"}`

**User impact:** "Find Related" goes from "73% similar" to "Shared ideas: [AI alignment] [mesa-optimization]". Users understand WHY articles are connected.

**Dependencies:** OpenAI API (already integrated), existing Celery pipeline, no new infra beyond one table

**Risks:**
- Extracted tags may be too verbose (multi-word phrases that don't normalize well). Mitigation: prompt specifies max 3-word labels; validate length before inserting.
- Backfill cost. Mitigation: gpt-4o-mini, ~$0.0002/article; rate-limit to 50/min.

**Success criteria:**
- 80% of new articles have semantic tags within 60s of ingestion completing
- Semantic tag extraction failure rate < 5%
- "Find Related" click events tracked via telemetry
- Manual spot-check: tags are legible and specific for 15/20 sampled articles
- No regression in "Find Related" P95 latency

---

### Phase 2 (Weeks 3–4): Tag clustering and reading themes

**Goal:** A "Reading Themes" surface shows users the dominant themes in their library.

**Build:**
- Weekly Celery beat task `cluster_user_tags`: for each user, load all `auto_tags` + their embeddings from `tag_embeddings`; run DBSCAN on the embedding vectors (auto-selects cluster count); min cluster size = 3 articles
- `reading_clusters` table: `{id UUID PK, user_id, label TEXT, article_ids ARRAY(UUID), tag_labels ARRAY(TEXT), updated_at}`
- Cluster label: highest-frequency tag label within the cluster (rule-based; no LLM call unless top label is a single character or blank)
- New page `/themes`: card grid — cluster name, article count, top 3 article thumbnails + titles
- Link from library header: "N reading themes →" entry point
- EmptyState for users with < 10 articles with semantic tags

**User impact:** User can answer "what am I reading about?" in one click.

**Dependencies:** Phase 1 complete (`tag_embeddings` must be populated)

**Risks:**
- DBSCAN epsilon sensitivity. Mitigation: tune on a test corpus; log cluster count per run; alert if count drops > 50% week-over-week.
- Too few articles → no clusters. Mitigation: EmptyState until ≥10 articles with semantic tags.

**Success criteria:**
- `/themes` renders for all users with ≥10 semantic-tagged articles
- Clusters semantically coherent (manual review of 3 user libraries)
- Library page LCP within 10% of pre-phase baseline

---

### Phase 3 (Weeks 5–6): Tag-grouped highlight connections

**Goal:** ConnectionsPanel groups connections by shared tag rather than raw score ranking. Remove the arbitrary 5-per-highlight cap.

**Build:**
- Update `/search/connections/article/{content_id}` response: include `shared_tags: list[str]` per highlight pair
- Tag-based deduplication: if N highlight pairs share the tag "LTV payback", surface best 2 pairs and collapse the rest ("+ N more on this tag")
- Remove hardcoded 5-per-highlight cap; replace with per-tag grouping logic
- Frontend: ConnectionsPanel renders tag-labeled sections instead of flat article list
- Telemetry: tag group click, expand click

**User impact:** "Your highlights on [AI alignment] connect to 4 articles. Your highlights on [mesa-optimization] connect to 2 articles."

**Dependencies:** Phase 1 (semantic tags per article)

**Success criteria:**
- ConnectionsPanel renders tag-grouped sections without regressions
- P95 response time for article connections endpoint < 500ms
- ConnectionsPanel open rate stable or increasing

---

### Phase 4 (Weeks 7–8): Draft workspace reading integration

**Goal:** When writing a draft, the workspace proactively surfaces relevant reading from the user's library.

**Build:**
- `GET /drafts/{id}/relevant-reads`: extracts topic from draft title + first 200 chars; runs hybrid search against user's library; returns top 5 results with matched tag labels
- 50-word minimum guard: return `{items: []}`, not error
- Draft sidebar panel "Relevant Reading": 5 cards with article title, reading status badge, matched tags, estimated reading time
- Debounced refresh: trigger on draft save, 500ms after typing stops; cache result until topic shifts
- User scoping validation: only the authenticated user's library items

**User impact:** Writing a draft on AI alignment surfaces 4 saved articles to cite — without leaving the workspace.

**Dependencies:** Phase 1 (semantic tags), hybrid search (already built)

**Risks:**
- Noisy results for short drafts. Mitigation: 50-word minimum; show "based on draft topic: [detected topic]"
- Real-time performance. Mitigation: debounce + cache; hard timeout 1.5s → return empty gracefully

**Success criteria:**
- Relevant reads API P95 < 1s
- 20%+ of users with active drafts click at least one relevant-read result
- Zero cross-user data leakage (verified by test)

## 9) TDD build methodology

**Principle: vertical slices via tracer bullets.** One test → one implementation → repeat. Never write all tests for a phase first. Tests verify observable behavior through public interfaces only — they must survive internal refactors.

### Phase 1 tracer bullet

Proves the enhanced tag extraction pipeline end-to-end:

```python
def test_generate_tags_stores_semantic_labels(db, test_user):
    # Given: article with title + summary text
    item = create_content_item(db, test_user,
        title="The Alignment Problem",
        summary="Explores deceptive alignment and mesa-optimization in deep learning systems.")

    # When: enhanced generate_tags runs
    generate_tags(str(item.id))

    # Then: auto_tags has fine-grained labels (not just "ai")
    db.refresh(item)
    assert len(item.auto_tags) >= 3
    assert any(len(tag.split()) > 1 for tag in item.auto_tags)  # at least one multi-word tag
```

This proves: task → LLM extraction → `auto_tags` written. Once green, subsequent tests add edge cases.

### Phase 1 behaviors (prioritized)

1. Enhanced `generate_tags` writes 3-5 fine-grained labels to `auto_tags`
2. LLM failure leaves `auto_tags` unchanged (not corrupted with empty/invalid data)
3. Each new tag label is upserted into `tag_embeddings` with a valid embedding vector
4. Backfill task is idempotent — running twice does not duplicate tags or embeddings
5. `GET /search/{item_id}/similar` response includes `shared_tags: list[str]` per result
6. Articles with no `full_text` fall back to `title + description` without error
7. Cross-user isolation — user A's tags are not returned in user B's similarity results

### Phase 2 behaviors (prioritized)

1. `cluster_user_tags` produces ≥1 cluster for a user with 15 semantic-tagged articles
2. Task skips users with < 10 articles without raising an error
3. `GET /themes` returns clusters with `article_count` and top 3 article IDs
4. Cluster update is idempotent — rerun replaces existing clusters, does not append
5. `GET /themes` for user with 0 clusters returns `{clusters: []}`, not 404

### Phase 3 behaviors

1. `/search/connections/article/{id}` response includes `shared_tags` per highlight pair
2. Per-tag deduplication: > 2 pairs on same tag are collapsed (not returned flat)
3. P95 response time < 500ms for article with 20 highlights

### Phase 4 behaviors

1. `GET /drafts/{id}/relevant-reads` returns ≤5 items matching draft tags
2. Returns `{items: []}` for drafts < 50 words
3. User A cannot receive user B's library items in relevant-reads
4. Draft with zero matching tags returns empty list gracefully

### Deep module interface principle

Each new module: small public interface, all complexity hidden:

```python
# Good
async def generate_tags(item_id: str, db: Session) -> None:
    # LLM extraction, tag normalization, embedding upsert — all hidden

# Bad — leaks implementation
async def extract_semantic_labels_from_content(title, summary) -> list[str]: ...
async def upsert_tag_embeddings(labels, db): ...
```

Tests exercise only `generate_tags`. Prompt wording, retry logic, embedding batching are implementation details — tests must not break when those change.

### What NOT to test

- OpenAI API response structure
- pgvector distance calculation
- Internal prompt template wording
- Celery task routing internals

## 10) SOTA AI coding practices for this build

See also: CLAUDE.md "AI-assisted development practices" (applies to all features). This section covers connection-build-specific application of those practices.

### Build sequence per phase

```
1. TaskCreate for the phase
2. /plan <phase goal>          → implementation blueprint (Plan subagent)
3. TDD loop (inline):
   a. RED:   write one failing test
   b. GREEN: write minimal passing implementation
   c. Repeat for each behavior (prioritized list above)
   d. Refactor after all green
4. /pre-commit-dev             → lint + types + tests + ARCHITECTURE.md update
5. Single commit (one feature = one commit)
6. TaskUpdate → completed
7. If context getting long → /handoff
```

### Parallel execution for this feature

Phase 1 has independent backend and frontend work. Launch both in the same message:

```
Parallel agents:
  A: Enhance generate_tags task + create tag_embeddings table + migration
  B: Write semantic tag pill component for "Find Related" card UI

Sequential gate (wait for A):
  Wire B's component to A's shared_tags API response field
```

Agent B's prompt must include the `shared_tags: list[str]` response schema explicitly. It cannot rely on Agent A's context.

### Feature flags

```
NEXT_PUBLIC_SHOW_SEMANTIC_TAGS=false    # Phase 1 — tag pills in "Find Related"
NEXT_PUBLIC_SHOW_READING_THEMES=false   # Phase 2 — /themes page
NEXT_PUBLIC_SHOW_DRAFT_READS=false      # Phase 4 — draft sidebar
```

### Cost efficiency

- **Semantic tag extraction**: gpt-4o-mini, structured JSON output, max_tokens=150. ~$0.0002/article.
- **Tag embedding upsert**: embed once per unique label, reuse from `tag_embeddings`. Most labels recur across articles — embedding cost amortizes quickly.
- **Cluster labeling**: rule-based first (highest-frequency tag in cluster), LLM fallback only for ambiguous edge cases.
- **Backfill**: rate-limited 50 articles/min during initial deploy.

## 12) Risks and mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Semantic tags too verbose (5-word labels that don't normalize) | Medium | Prompt: max 3 words per label; validate on write; truncate on violation |
| LLM hallucinations in tag labels | Medium | Structured JSON output mode; validate: non-empty strings, reject schema violations |
| Tag label collisions (similar ideas expressed differently) | Low | `tag_embeddings` enables fuzzy matching even without exact label match; low priority for Phase 1 |
| DBSCAN epsilon sensitivity | Low | Log cluster count per run; alert if drops > 50% week-over-week; tune in staging |
| Draft integration performance | High | Debounce 500ms; cache last result until topic shifts; hard 1.5s timeout → empty response |
| OpenAI cost spike during backfill | Medium | Rate limit 50/min; monitor spend; alert at 2× baseline daily cost |
| Semantic tags overwrite coarse tags users expect | Medium | Run extraction as a separate pass; do not clear existing user-confirmed `tags` |

## 13) Positioning and strategic outcome

**Before this work:**
Tags are coarse categorical labels. Connections show a similarity percentage with no explanation. Both features exist but don't compound into anything.

**After this work:**
Tags are the semantic layer. They explain connections ("Shared ideas: [AI alignment]"), power reading themes ("what am I actually reading about?"), and surface relevant reading while you write. The reading queue becomes organized knowledge.

**The moat:**
Tags-as-semantic-unit means every improvement to tag extraction improves connections, themes, and draft integration simultaneously. One investment, three surfaces. This is the "institutional memory" arc from `sedi-knowledge-system-roadmap.md` — this plan is Phase 0: making the semantic layer real before building the full knowledge graph.
